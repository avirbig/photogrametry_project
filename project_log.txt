ENTRY 0  -  PROJECT GOAL AND THE BIG IDEA          [date: 2026-07-06]
================================================================================

WHAT WE ARE TRYING TO DO
------------------------
We have 14 old photographs of the same building, taken from different
positions. By hand we marked "matching points" - the same physical feature
(a window corner, a roof edge) clicked in every photo where it is visible.

From those 2D pixel marks ALONE, we want to recover four things:
  1. Where each marked point sits in 3D space  -> its (X, Y, Z) coordinates.
  2. Where each camera was  -> the 3D position each photo was taken from.
  3. Which way each camera pointed  -> its 3D orientation.
  4. How much to trust each result  -> an error value for every point/camera.

At the end, everything is expressed relative to the marked points themselves:
the points sit at the centre of the coordinate system and the cameras are
placed around them. This whole family of techniques is called
"photogrammetry" or "Structure from Motion" (SfM) = working out 3D structure
from a set of ordinary photos.

THE ONE IDEA BEHIND EVERYTHING
------------------------------
A camera takes a 3D point and flattens it onto a flat picture. If you knew
exactly where the camera was, which way it pointed, and how zoomed-in it was,
you could PREDICT the exact pixel where any 3D point would land.

The reverse of that prediction is our entire task: starting from the pixels we
clicked, work backwards to recover the 3D points and every camera's position,
orientation, and zoom.

The compact way this is written (we will unpack every symbol later) is:

      x  ~  K [ R | t ] X

  X = the 3D point            (what we want)
  x = the pixel it lands on    (what we measured by clicking)
  R = camera orientation       (which way it points)  (what we want)
  t = tied to camera position  (where it sits)         (what we want)
  K = the camera's internal settings (mainly the zoom / focal length)

Don't worry about this line yet. Each piece gets its own entry with real
numbers when we reach the step that uses it.

ENTRY 7  -  STEP 3: THE CAMERA SETTINGS MATRIX K   [date: 2026-07-06]
================================================================================

GOAL OF THIS STEP
-----------------
Set up the "camera model" - a small table of numbers, called K, that describes
each camera's INTERNAL settings. We build one K per photo (14 in total). This
step SOLVES NOTHING yet; it only labels which camera settings we already know
and which single setting stays unknown. Every later step plugs K in.

FIRST, THREE PLAIN-WORDS IDEAS WE NEED
--------------------------------------
1) Pixel coordinates (x, y).
   x = how far RIGHT from the left edge, y = how far DOWN from the top edge,
   both counted in pixels. Top-left of the image is (0, 0).

2) Focal length, written f.
   A single number that captures "how zoomed-in" the lens is. A long lens
   (telephoto) has a big f; a wide lens has a small f. IMPORTANT: here f is
   measured in PIXELS, not millimetres, because we are working directly with
   pixel positions. Old photos never recorded it, so f is our unknown.

3) The principal point (cx, cy).
   The pixel where the lens axis pierces the image - essentially the CENTRE of
   the picture. Standard, safe assumption for old photos: it is the middle of
   the image, so  cx = width / 2  and  cy = height / 2.

WHAT K LOOKS LIKE
-----------------
K is a 3-row by 3-column table (a "3x3 matrix"). For our assumptions it is:

            column 1   column 2   column 3
          +--------------------------------+
   row 1  |    f          0         cx     |
   row 2  |    0          f         cy     |
   row 3  |    0          0          1     |
          +--------------------------------+

Reading every entry, one at a time:
   (row1,col1) = f   : horizontal zoom, in pixels.
   (row2,col2) = f   : vertical zoom. We put the SAME f in both because we
                       assume "square pixels" (a pixel is as wide as it is
                       tall), so the zoom is identical in x and y.
   (row1,col2) = 0   : "skew". Would be non-zero only if the pixel grid were
                       slanted (parallelogram-shaped pixels). Real cameras have
                       square grids, so this is 0.
   (row1,col3) = cx  : horizontal position of the image centre.
   (row2,col3) = cy  : vertical position of the image centre.
   (row3,col1) = 0
   (row3,col2) = 0
   (row3,col3) = 1   : a fixed bookkeeping 1 (see the note on the "extra 1"
                       below). It is always 1 and never changes.

WHAT K DOES (why this table is shaped like this)
------------------------------------------------
K's job is to convert a direction the camera is looking along into the pixel
that direction lands on. If a 3D point sits at camera-relative position
(Xc, Yc, Zc) - where Zc is "depth", how far in front of the camera it is - then
the pixel it projects to is:

        x  =  f * (Xc / Zc)  +  cx
        y  =  f * (Yc / Zc)  +  cy

In words: take the sideways/up-down angle to the point (that is the divide-by-
depth part, Xc/Zc and Yc/Zc), multiply by the zoom f to turn the angle into a
pixel offset, then shift by the centre (cx, cy) because pixel (0,0) is the
top-left corner, not the middle. The K table above is just those two lines
written in one compact block so we can reuse it with matrix multiplication.

A NOTE ON THE "EXTRA 1" (homogeneous coordinates)
-------------------------------------------------
You will see pixels written as three numbers (x, y, 1) instead of two, and 3D
points as four numbers (X, Y, Z, 1). That trailing 1 is a standard trick called
"homogeneous coordinates". It lets a single matrix multiplication do both the
multiply-by-f AND the add-the-centre in one clean step (adding a constant is
otherwise not something plain matrix multiply can do). Treat the 1 as harmless
padding for now; it is why K's bottom-right entry is 1. We will use it live at
Step 4.

OUR ASSUMPTIONS, LISTED (per the guide, Step 3)
-----------------------------------------------
  [known]    principal point = image centre  -> cx = width/2, cy = height/2
  [known]    square pixels                    -> one shared f, both diagonal
  [known]    no skew                          -> the off-diagonal 0
  [UNKNOWN]  focal length f                   -> different and unknown per photo,
                                                 estimated later (Steps 4 & 7)

WHAT THIS STEP PRODUCED
-----------------------

WHY WE DON'T JUST GUESS f
-------------------------
We could guess (a common rough first guess is f approximately equal to the
image width in pixels), but a wrong zoom bends the whole reconstruction. The
method's strength is that it works f out FROM THE MARKED POINTS, so we keep it
honestly unknown here and let the geometry determine it.

ENTRY 8  -  STEP 4: THE SEED PAIR (RELATIVE POSE)  [date: 2026-07-06]
================================================================================

GOAL OF THIS STEP
-----------------
Take two photos and work out how the SECOND camera is positioned and rotated
relative to the FIRST, using only their shared marked points. This "seed pair"
is the foundation the whole 3D reconstruction is built on. The two outputs are:
      R = the second camera's ROTATION (which way it points)
      t = the second camera's TRANSLATION direction (which way it sits)

WHICH TWO PHOTOS? (a change from the original plan)
---------------------------------------------------
Entry 3 suggested 02_front_wide + 04_front_elevated because they share the most
points (25). But when we actually ran the math, that pair behaved badly:

  - A building facade is nearly FLAT. The guide warns (Common Mistakes) that
    points on one flat surface make the math unstable. We tested flatness by
    fitting a "homography" (a flat-scene model): if it fits as well as the full
    3D model, the scene is effectively flat. For 02+04 it fit too well.
  - Worse, the focal-length recovery was unreliable for that pair (see below).

So we scanned ALL photo pairs and scored each on (a) NON-flatness and (b) how
well-conditioned the focal recovery is. The winner:

      SEED PAIR = 01_front_clean.jpg  +  04_front_elevated.jpg
      - 24 shared points (plenty; only 8 are required)
      - clearly non-flat (the flat-scene model fits poorly, as we want)
      - 24 out of 24 points end up correctly in front of both cameras

  Camera 1 (the reference, placed at the origin) = 01_front_clean.jpg
  Camera 2 (the one we solve for)                = 04_front_elevated.jpg

THE FOCAL-LENGTH PROBLEM (an honest, report-worthy finding)
-----------------------------------------------------------
Recall from Step 3 that the focal length (zoom) f is unknown. In principle the
seed pair can estimate it. In practice, TWO views of a building do NOT pin it
down: when we searched for the focal that best fits, the value ran off toward
infinity instead of settling. This is a real limitation, not a bug - two
photos simply do not contain enough depth information to fix the zoom reliably.

DECISION: we do not pretend to estimate f here. Instead we FIX a sensible
starting value - a "normal lens" assumption:

      f (in pixels) = 1.2 * (the larger of the image's width, height)

giving  fA = 1147 px  (for 01_front_clean, 956x683)
        fB =  883 px  (for 04_front_elevated, 736x444).

These are only STARTING values. Step 7 (bundle adjustment) refines the focal
lengths later, using all cameras together, where they ARE well-constrained.
With these focals the two camera settings matrices (Step 3) become:

              [ 1147.2    0     478.0 ]           [ 883.2    0    368.0 ]
   K1 =       [    0   1147.2   341.5 ]   K2 =     [   0   883.2   222.0 ]
              [    0      0       1.0 ]           [   0     0       1.0 ]

THE FIVE STAGES - HOW THE MATH WORKS (the concepts)
---------------------------------------------------
Step 4 is carried out in five calculation stages. This part explains what each
stage does and why; the next part runs them on the real numbers.

Two small tools used throughout:
  - Homogeneous pixel (from Entry 7): a pixel (x, y) is written (x, y, 1).
  - Dot product: multiplying a row by a column means "multiply matching entries
    and add them up", e.g. (a,b,c) . (d,e,f) = a*d + b*e + c*f. The small "^T"
    (transpose) just turns a column into a row so this can be done.

The master equation. For one physical point seen in both photos - pixel x1 in
photo 1 and pixel x2 in photo 2 - there is a 3x3 table F (the FUNDAMENTAL
MATRIX) for which:

        x2^T F x1 = 0

Plain meaning: F captures the geometric relationship between the two camera
viewpoints. If x1 and x2 really are the same feature, the equation gives 0; a
mismarked point gives a non-zero "how-wrong" score. The task of Step 4 is to
find the F that best satisfies this for all the shared points, then unpack the
camera motion hidden inside it. The five stages do exactly that.

Stage 1 - One equation per matched point.
   F has 9 unknown entries (f11 ... f33). Writing x2^T F x1 = 0 out in full for
   a pair (u1,v1)<->(u2,v2) gives a single linear equation whose unknowns are
   the f's and whose coefficients are just products of the measured pixels:

     u2u1 f11 + u2v1 f12 + u2 f13 + v2u1 f21 + v2v1 f22 + v2 f23
                                              + u1 f31 + v1 f32 + f33 = 0

   Each shared point contributes one such row. Stacking all rows gives a tall
   table A, and the whole problem becomes:  A f = 0  (find the 9 numbers of F
   that make every row come out about 0).

Stage 2 - Solve A f = 0 with SVD.
   Ordinary division is no help here (the useless answer f = all-zeros would
   "solve" it). The tool is SVD (SINGULAR VALUE DECOMPOSITION), which splits any
   matrix into three simple moves - rotate, then scale each axis, then rotate -
   and reports the scale amount (the "singular value") for every direction. We
   want the direction that A shrinks CLOSEST TO ZERO (smallest singular value):
   that direction is the choice of F that disagrees with the point matches the
   least - the best fit. Reshaping those 9 numbers into a 3x3 grid gives a draft
   of F.

Stage 3 - Clean up F (enforce "rank 2").
   Because clicks are slightly imprecise, the draft F is a little off. A genuine
   fundamental matrix must obey one extra rule (called rank 2: its determinant
   must be exactly 0), which keeps the geometry self-consistent. The fix uses
   SVD again: list F's three scale amounts and force the smallest to 0, e.g.
   diag(5.0, 3.0, 0.02) -> diag(5.0, 3.0, 0), then rebuild F. ("diag" means a
   matrix with those numbers on its diagonal and 0 everywhere else.) The tiny
   value removed was pure noise.

Stage 4 - Focal length and the essential matrix E.
   F is expressed in raw pixels and still hides the zoom. We convert it to the
   ESSENTIAL MATRIX E, which folds the Step-3 camera settings K back in:

        E = K2^T F K1

   (K1, K2 are the two cameras' settings matrices from Entry 7.) E is the same
   relationship as F but with the pixel/zoom part removed, leaving pure position
   and rotation - which is what lets us extract the camera motion. Because the
   focal length is unknown, we must choose a value to use here; the section
   "THE FOCAL-LENGTH PROBLEM" above explains the choice we made for this data.

Stage 5 - Pull the rotation R and translation t out of E.
   E secretly bundles two things together: R (camera 2's rotation) and t
   (camera 2's translation direction), combined as E = [t]x R. We unpack them
   with SVD one last time, E = U S V^T, together with a FIXED helper matrix

        W = [ 0  -1   0 ]
            [ 1   0   0 ]
            [ 0   0   1 ]

   giving:   R = U W V^T        t = the last column of U

   This produces FOUR candidate (R, t) answers (two rotations x two signs of t).
   Only one places the 3D points IN FRONT of both cameras; the other three put
   them behind, which is physically impossible. Testing this - the CHEIRALITY
   CHECK - selects the single correct answer.

That is the whole method. The next part carries out these five stages on the
real seed pair.

WHAT R AND t REPRESENT  (the plain-words meaning)
-------------------------------------------------
Set up a coordinate frame on camera 1: it sits at the origin (0,0,0) and looks
straight down the +Z axis into the scene. In this frame, +X points to the
right, +Y points down (matching image rows), +Z points forward into the scene.

R - THE ROTATION OF CAMERA 2.
   R is a 3x3 rotation matrix: it says how camera 2 is TURNED compared with
   camera 1. Every rotation matrix can be summarised by a single angle - how
   many degrees it turns. Here that angle is 13.0 degrees. Meaning: camera 2
   (04_front_elevated) is aimed about 13 degrees away from camera 1's
   direction - a modest change of viewpoint, which fits two fairly frontal
   photos of the same facade. If R were the identity matrix, the two cameras
   would point in exactly the same direction (0 degrees).

t - THE TRANSLATION (WHERE CAMERA 2 SITS).
   t is a direction arrow with three numbers. Strictly, t is the "translation
   part" of camera 2's pose; the actual PHYSICAL POSITION of camera 2 (its
   centre) is computed from R and t as:

         C2  =  -R^T t   =   ( -0.40,  -0.05,  -0.91 )

   Read in our axes (X right, Y down, Z forward): camera 2 sits mostly BACK
   along the viewing axis (-Z) and a bit to the LEFT (-X), barely up or down.
   In plain words: relative to camera 1, the second shot was taken from a
   position offset backward-and-left, then re-aimed by 13 degrees.

   IMPORTANT - only the DIRECTION is known, not the distance. t is written as a
   unit arrow (length 1) because two photos cannot reveal the true separation
   between the cameras (this is the same "scale is arbitrary" fact from Entry
   5). We know WHICH WAY camera 2 is displaced, not by how many metres. The
   whole model's size is fixed only later, and only if a real distance is
   known (which we do not have).

   Camera 1 itself is, by definition, R = identity and t = (0,0,0): it is the
   anchor everything else is measured against.
